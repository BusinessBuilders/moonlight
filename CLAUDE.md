## Platform Overview

Build a **production-grade, self-hosted, multi-tenant CMS platform** that:

1. Lets each Business Builders **client** log into their own branded admin panel and edit their site content
2. **Plugs into any React/Next.js site** via a single NPM package or lightweight API connection
3. **Removes cleanly** if a client stops paying or migrates hosting — zero residual code, zero orphaned dependencies
4. Is **100% self-hosted** by Business Builders on their own infrastructure, with no per-seat SaaS fees

## Architecture: How It Works

```
Business Builders Server (self-hosted)
│
├── Platform Admin (Payload CMS)
│   ├── Client A tenant → their collections, media, settings
│   ├── Client B tenant → their collections, media, settings
│   └── Client C tenant → ...
│
└── API Endpoints (auto-generated by Payload)
    ├── /api/client-a/[collection]
    ├── /api/client-b/[collection]
    └── /api/client-c/[collection]

Client's React/Next.js Site (any hosting)
│
└── @business-builders/cms-client (NPM package)
    ├── Fetches content from Business Builders API
    ├── Provides React hooks: usePage(), useGallery(), useServices(), etc.
    └── Removes cleanly: uninstall package → site falls back to static content
```

**Key principle:** The client's site never depends on a database — it fetches content from the Business Builders API. If the client stops paying, the API key is revoked and the site falls back to static/default content gracefully.

## Platform Features

### Multi-Tenant Payload CMS Setup

Each client gets:
- **Isolated tenant** in Payload with their own collections
- **Scoped admin login** — clients only see their own data, never other clients
- **Customizable collections** per client based on their site type (farm, restaurant, contractor, etc.)
- **Role system:**
  - `super_admin` — Business Builders team (full access to all tenants)
  - `client_admin` — Client owner (edit their own content only)

## NPM Integration Package — `@business-builders/cms-client`

This is the key portability mechanism. Build it as a separate package in the monorepo.

### Installation on client site:
```bash
npm install @business-builders/cms-client
```

### `.env` on client site:
```env
BB_API_URL=https://cms.business-builder.online
BB_API_KEY=client_xxxxxxxxxxxx
BB_TENANT_ID=moonlight-farm
```

### Usage in client's Next.js site:
```tsx
import { useSiteSettings, useGallery, useServices, useEvents } from '@business-builders/cms-client'

// Fetch site settings
const { data: settings } = useSiteSettings()

// Fetch gallery images
const { data: gallery } = useGallery({ category: 'cattle' })

// Fetch services
const { data: services } = useServices()
```

### Clean Removal:
```bash
npm uninstall @business-builders/cms-client
```
- Remove the 3 env variables
- Site falls back to static content — no broken imports, no orphaned API calls
- Zero dead code remains

**Implementation requirements:**
- All hooks return `{ data, loading, error }` pattern
- Include TypeScript types for all returned data
- Graceful degradation: if API is unreachable, return empty arrays / null (never crash)
- Optional: `fallbackData` prop on each hook for static defaults

## API Security

### API Key System
- Each client gets a unique API key on signup (`BB_API_KEY`)
- Keys are scoped to their tenant ID
- Keys can be revoked instantly by Business Builders (stops content delivery immediately)
- Keys stored hashed in PostgreSQL (never in plaintext)

### Per-Tenant Rate Limiting
- Implement rate limiting per API key (e.g., 500 requests/min)
- Use Redis or in-memory rate limiter

### Rotation
- Business Builders can rotate any client's API key from the super admin panel
- Client receives new key via email (Resend)

## Portability & Hosting Migration

### For clients who want to move hosting:
The client's **React site** can move to any host (Vercel, Netlify, self-hosted VPS) without issue — it only needs the 3 env variables pointing to Business Builders' API. No database migration needed on their end.

### For Business Builders to move their own infrastructure:
- `docker-compose.yml` includes all services (Next.js app, PostgreSQL, Redis)
- Payload migrations handle database schema
- Cloudinary or S3 for media (portable)
- New server: `docker compose up` — fully running

### For a client who cancels and takes their site in-house:
1. Export their content via Payload's built-in export
2. Client installs their own Payload instance (open source — no licensing cost)
3. Import their content
4. Update 3 env variables to point to their own instance
5. Business Builders removes their tenant

## Client Onboarding Flow (Automated)

When Business Builders adds a new client:

1. Super admin fills form: business name, owner email, site type (farm / contractor / restaurant / etc.)
2. Platform auto-creates:
   - New tenant record in PostgreSQL
   - Default collections based on site type template
   - Client admin login credentials
   - API key
3. Welcome email sent to client (via Resend) with:
   - Admin panel URL
   - Temporary password
   - API key + integration instructions
4. Client logs in, changes password, starts editing content
